## 二、代码规范

### 2.1 命名规范
- **类名**: PascalCase, 如 `PurReceiptService`
- **方法名**: camelCase, 如 `addProduct`, `getUserById`
- **变量**: camelCase, 如 `supplierName`
- **常量**: UPPER_SNAKE_CASE, 如 `STATUS_DRAFT`
- **包名**: 全小写, 如 `com.industrial.erp.modules.purchase.service`
- **表名**: snake_case, 如 `pur_receipt_detail`
- **列名**: snake_case, 如 `total_amount_tax`

### 2.2 类注释模板
每个类必须有 Javadoc:
```java
/**
 * 采购入库服务
 * 业务: 生成入库单 -> 写明细 -> 调用 StockService 库存入库 -> 生成应付台账
 *
 * @author ERP Team
 * @since 1.0.0
 */
@Service
@RequiredArgsConstructor
public class PurReceiptService {
    ...
}
```

### 2.3 异常使用规范
- **业务异常**: 抛 `BizException`, 由 GlobalExceptionHandler 统一处理
- **不要** 捕获后吞掉异常
- **不要** 用 `System.out.println`, 用 `log.info/warn/error`

```java
// 正确
if (stock == null) {
    throw new BizException("库存不存在");
}

// 错误
if (stock == null) {
    System.out.println("库存不存在");
    return null;
}
```

### 2.4 事务规范
- 写操作加 `@Transactional(rollbackFor = Exception.class)`
- 禁止在 Controller 调用 Service 后, 多个写操作不加事务
- 跨服务调用 (Feign) 加 `@Transactional(rollbackFor = Exception.class, propagation = Propagation.REQUIRES_NEW)`

### 2.5 权限规范
- Controller 入口加 `permService.requirePerm("module:action")`
- 例: `purchase:receipt:add`, `inventory:stock:list`
- 与前端 `v-permission` 指令配套

### 2.6 审计字段自动填充
- 通过 MyBatis-Plus `MetaObjectHandler` 自动填充 `create_by / create_time / update_by / update_time`
- 业务代码**不要**手动设置这些字段

## 三、核心模式与最佳实践

### 3.1 单据状态机
单据常见状态流转:
```
DRAFT (草稿) --check--> CHECKED (已审核) --finish--> FINISHED (已完成)
   |                                                       |
   |--cancel--> CANCELLED                                  CLOSED (已关闭)
```

实现模式:
```java
@Transactional(rollbackFor = Exception.class)
public void check(Long id) {
    XxxBill bill = billMapper.selectById(id);
    if (bill == null) throw new BizException("单据不存在");
    if (!Constants.STATUS_DRAFT.equals(bill.getBillStatus())) {
        throw new BizException("只有草稿状态可审核");
    }
    // 业务处理...
    
    // 更新状态
    XxxBill upd = new XxxBill();
    upd.setId(id);
    upd.setBillStatus(Constants.STATUS_CHECKED);
    billMapper.updateById(upd);
}
```

### 3.2 库存操作 (重要!)

**所有库存操作必须通过 StockService, 不能直接 update inv_stock 表!**

```java
// 错误的写法: 直接 SQL
@Update("UPDATE inv_stock SET qty = qty - #{qty} WHERE id = #{id}")
int updateQty(...);

// 正确的写法: 通过服务
stockService.outStock("SAL_DELIVERY", billId, billNo, detailId,
        warehouseId, warehouseName, locationId, locationName,
        productId, unitId, unitName, batchNo,
        qty, price, sourceNo, customerId, supplierId, remark);
```

StockService 内部:
- 加 Redis 分布式锁 (防并发)
- 加 MySQL 行锁 `SELECT...FOR UPDATE`
- 校验库存是否足够 (防负库存)
- 写流水 `inv_ledger`
- 更新 `inv_stock` (自动重算移动加权平均成本)
- 更新商品主表 `base_product.cost_price`

### 3.3 单据号生成
使用 `BillNoGenerator`:
```java
String billNo = billNoGenerator.generate(Constants.BILL_RKP);
// 格式: RKP202606160001 (prefix + yyyyMMdd + 4位流水)
```

底层用 Redis `INCR` 原子生成, 每日 0 点重置。

### 3.4 分页查询
- 统一参数: `pageNum`, `pageSize`
- 返回 `PageResult<T>`:
```java
@GetMapping("/page")
public R<PageResult<MyEntity>> page(@RequestParam(defaultValue = "1") Integer pageNum,
                                    @RequestParam(defaultValue = "20") Integer pageSize,
                                    @RequestParam(required = false) String keyword) {
    return R.ok(PageResult.of(myService.page(pageNum, pageSize, keyword)));
}
```

### 3.5 操作日志
在 Controller 方法上加 `@OperLog` 注解:
```java
@PostMapping
@OperLog(module = "采购管理", businessType = "ADD")
public R<Void> add(@RequestBody PurReceipt receipt) {
    service.add(receipt);
    return R.ok();
}
```

AOP 会自动:
- 记录模块、业务类型、方法、URL、参数、IP、用户、耗时、异常

### 3.6 数据权限
通过 `sys_role.data_scope` 控制:
- 1 全部  2 本部门及下级  3 本部门  4 本人
- Service 中调用 `permissionService.getDataScope()` 拿到当前用户的数据范围
- 在 SQL 中根据 dataScope 拼条件

简单实现可在 Service 中:
```java
if (dataScope == 4) {
    w.eq(XxxBill::getCreateBy, userId);
} else if (dataScope == 3) {
    w.eq(XxxBill::getDeptId, deptId);
}
```

## 七、MyBatis-Plus 用法

### 7.1 BaseMapper 基础 CRUD
```java
userMapper.insert(user);              // 新增
userMapper.updateById(user);          // 修改
userMapper.selectById(id);            // 详情
userMapper.deleteById(id);            // 删除 (逻辑)
userMapper.selectList(wrapper);       // 列表
userMapper.selectPage(page, wrapper); // 分页
```

### 7.2 复杂查询
```java
LambdaQueryWrapper<PurReceipt> w = new LambdaQueryWrapper<>();
w.like(StrUtil.isNotBlank(billNo), PurReceipt::getBillNo, billNo);
w.eq(supplierId != null, PurReceipt::getSupplierId, supplierId);
w.ge(startDate != null, PurReceipt::getBillDate, startDate);
w.orderByDesc(PurReceipt::getId);
return mapper.selectPage(page, w);
```

### 7.3 自定义 SQL (XML)
```java
public interface InvStockMapper extends BaseMapper<InvStock> {
    /**
     * 加锁查询 (悲观锁), 用于出库
     */
    InvStock selectForUpdate(@Param("warehouseId") Long warehouseId,
                             @Param("locationId") Long locationId,
                             @Param("productId") Long productId,
                             @Param("batchNo") String batchNo);
}
```

XML:
```xml
<select id="selectForUpdate" resultMap="StockMap">
    SELECT * FROM inv_stock
    WHERE warehouse_id = #{warehouseId}
      AND IFNULL(#{locationId}, 0) = IFNULL(location_id, 0)
      AND product_id = #{productId}
      AND IFNULL(#{batchNo}, '') = IFNULL(batch_no, '')
      AND deleted = 0
    FOR UPDATE
</select>
```

## 十、常见 Bug 与规避

### 10.1 循环依赖
A 依赖 B, B 依赖 A 会导致启动失败。
解决: 重构或使用 `@Lazy` 注入。

### 10.2 事务失效
场景: Controller 调 Service A, A 调 Service B, B 抛异常
解决: 事务边界要在调用入口, 不要拆得过于零散

### 10.3 N+1 查询
场景: 列表查询, 每行再查一次关联表
解决: 用 `selectMaps` 一次性 JOIN, 或在 Service 中批量查

### 10.4 BigDecimal 比较
```java
// 错误: ==
if (a == b) { ... }

// 正确: compareTo
if (a.compareTo(b) == 0) { ... }
if (a.compareTo(BigDecimal.ZERO) > 0) { ... }
```

### 10.5 浮点数
不要用 `double` 表示金额, 必须用 `BigDecimal`。

### 10.6 时区
数据库和 JVM 都配置 `Asia/Shanghai`:
```yaml
spring:
  jackson:
    time-zone: GMT+8
  datasource:
    url: ...?serverTimezone=Asia/Shanghai
```

### 10.7 大数据量导出
- 不要一次查全部到内存
- 用 EasyExcel 的 `Page` 分批写
- 超过 10w 行用 SXSSF

## 十一、开发工作流

### 11.1 新需求开发
1. 确认需求, 拉相关人员评审
2. 在 `feature/` 分支开发
3. 数据库 DDL 提交到 `sql/migrations/`
4. 后端代码: entity -> mapper -> service -> controller
5. 前端代码: api -> view
6. 单元测试
7. 自测, 提 PR
8. Code Review
9. 合并到 develop

### 11.2 修复 Bug
1. 在 issue 中描述复现步骤
2. 分支命名: `fix/issue-123-bug-desc`
3. 修复后增加测试用例
4. PR 描述中关联 issue

### 11.3 性能优化
1. 用 `EXPLAIN` 分析慢 SQL
2. 必要时加索引
3. 引入缓存
4. 异步处理 (MQ / @Async)
5. 分库分表 (极端情况)
