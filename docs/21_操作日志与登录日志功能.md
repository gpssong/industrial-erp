# 21 操作日志与登录日志功能 (2026-07-06)

## 概述

为响应用户审计需求,本次会话在系统设置页加入了**操作日志(含删除数据可见)** + **登录日志** 两个新 tab,顺便修复了若干隐藏 bug。

**用户原始诉求**:"我需要在系统设置页面添加一个日志功能,能让我了解每个账号进行的操作。被删除的信息也要在这里看到。"

---

| 端 | 入口 |

| | - 操作日志子 tab: 模块/业务/操作人筛选 + 删除快照 + 请求参数 + 返回结果弹窗 + 清理 90 天前 |

## 设计决策

| 决策 | 选项 | 选择理由 |
|---|---|---|
| 删除方式 | 物理删除 / 软删除 | **软删除**(`mapper.update set deleted=1`)。便于审计 + 误删恢复 |
| 前端位置 | 独立路由 / 系统设置 tab | **SystemSettings tab 内嵌** — 不加菜单 + 不增权限 seed |
| 截到什么程度 | add/update/全部 / 仅删除快照 | **仅删除快照** — 改动最小,符合"看被删除"诉求 |
| SysLoginLog | 顺便补 / 不做 | **顺便补完整** — 原 controller 是空壳 |

## 联调时发现并修复的 bug

### Bug 1 (HIGH): `OperLogPublisher.publishDeleteSnapshot` 序列化失败被 catch 吞掉, snapshot_json 永远是 NULL

**根因**: 在 aspect 里 `new ObjectMapper()` 拿了一个**裸 ObjectMapper**, 不带 JSR310 模块,
无法序列化 `java.time.LocalDateTime` 字段 (如 `BaseCustomer.createTime`)。
抛 `InvalidDefinitionException`,但被 `catch (JsonProcessingException ignore) {}` 吞掉了,
所以 snapshot_json 列永远是 NULL。

**修复** (`backend/src/main/java/com/industrial/erp/modules/system/aspect/OperLogPublisher.java`):
构造器注入 Spring 配置好的 ObjectMapper:

```java
public OperLogPublisher(ApplicationEventPublisher publisher, ObjectMapper om) {
    this.publisher = publisher;
    this.om = om;  // Spring 全局 OM, 自带 JSR310
}
```

**验证**: 删除测试客户后 `sys_oper_log.snapshot_json` 含完整 entity JSON。

### Bug 2 (HIGH): `SalOrderService.delete` 不级联删 detail 表

**根因**: 原代码只 `orderMapper.deleteById(id)`, `sal_order_detail` 保留,
留下孤儿记录且后续查询可见。

**修复**: 主表+子表一并软删,加 `@Transactional` 包裹。

### Bug 3 (MED): `OperLogAspect.saveLog` 标 `@Async` 但被同一类 `around()` 自身调用,代理链不生效

**根因**: Spring AOP 不会拦截同一类内部的调用, `@Async` 不生效,
saveLog 实际是同步执行,阻塞业务线程。

**修复**: 改用 `ApplicationEventPublisher.publishEvent(new OperLogEvent(...))` +
独立的 `@Async @EventListener` 接收。

### Bug 4 (LOW): 旧的 `OperLogAspect` 入口 `@annotation(OperLog)` 全代码库 0 引用

**根因**: 切面定义了但从未在 controller/service 上标注,日志永远是 0 条。

**修复**: 这次没改这个机制 — 现在改成"service.delete() 显式调用 `operLogPublisher.publishDeleteSnapshot`",
而不是依赖注解扫描。好处是**显式 > 隐式**,代码可读性强;坏处是 add/update 仍未记录。
**未来**: 若需要 ADD/EDIT 日志,可在 controller 的 add/update 方法上加 `@OperLog` 注解。
