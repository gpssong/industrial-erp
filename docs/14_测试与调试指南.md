### 2.3 测试约定

| 维度 | 规范 |
|---|---|
| 框架 | JUnit 5 (`org.junit.jupiter.api`) |
| 断言 | 优先 `assertEquals(expected, actual, delta)` 比较 `BigDecimal`,**禁用 `==`** |
| 命名 | `方法名_场景_期望结果`(`createDelivery_creditExceed_fail`) |
| 注解顺序 | `@Test` → `@Order` → `@Transactional` → `@Rollback` |
| 事务 | 默认 `@Transactional @Rollback`,**集成测试才提交** |
| 数据隔离 | 每次 `@Transactional` 自动回滚,不污染数据库 |
| 性能 | 单元测试 < 1s,集成测试 < 10s,E2E < 5min |
| 覆盖率 | Jacoco 报告,核心 Service 覆盖率 ≥ 70% |

## 四、并发与性能测试

### 4.1 库存并发压测

**JMeter 脚本**:50 并发,每个并发下单一笔,验证**无超卖、无负库存**。

```java
// StockConcurrencyTest.java
@SpringBootTest
class StockConcurrencyTest {

    @Autowired StockService stockService;
    static final int CONCURRENCY = 50;
    static final BigDecimal PER   = new BigDecimal("1");
    static final BigDecimal INIT  = new BigDecimal("30");

    @Test @Transactional @Rollback
    void noOverSell() throws Exception {
        // 准备 30 库存
        stockService.inStock("INIT", 1L, "INIT", 1L, 1L, "W1", null, null,
                1L, 1L, "kg", "B1", INIT, new BigDecimal("10"), "INIT", null, null, "init");

        ExecutorService pool = Executors.newFixedThreadPool(CONCURRENCY);
        CountDownLatch start = new CountDownLatch(1);
        AtomicInteger ok = new AtomicInteger();
        AtomicInteger fail = new AtomicInteger();

        for (int i = 0; i < CONCURRENCY; i++) {
            pool.submit(() -> {
                try {
                    start.await();
                    stockService.outStock("SAL", 1L, "C", 1L, 1L, "W1", null, null,
                            1L, 1L, "kg", "B1", PER, new BigDecimal("15"), "C", null, 1L, "test");
                    ok.incrementAndGet();
                } catch (BizException e) { fail.incrementAndGet(); }
                catch (Exception e) { fail.incrementAndGet(); }
            });
        }
        start.countDown();
        pool.shutdown(); pool.awaitTermination(30, TimeUnit.SECONDS);

        assertEquals(30, ok.get(),  "30 单应成功");
        assertEquals(20, fail.get(), "20 单应失败(库存不足)");
    }
}
```

### 4.2 慢 SQL 定位

```yaml
# application-dev.yml
mybatis-plus:
  configuration:
    log-impl: org.apache.ibatis.logging.nologging.NoLoggingImpl
```

或者:

```java
@Component
@Aspect
public class SqlTimingAspect {
    @Around("execution(* org.springframework.jdbc.core.JdbcTemplate.*(..))")
    public Object time(ProceedingJoinPoint pjp) throws Throwable {
        long t = System.currentTimeMillis();
        Object r = pjp.proceed();
        long cost = System.currentTimeMillis() - t;
        if (cost > 200) log.warn("Slow SQL {}ms: {}", cost, pjp.getArgs()[0]);
        return r;
    }
}
```

### 4.3 性能调优清单

| 现象 | 排查方向 |
|---|---|
| 接口 P99 > 1s | 1) 慢 SQL → 加索引 2) N+1 → 联表 / 批量查询 3) 大字段 → 拆分 |
| CPU 飙高 | 1) 死循环 2) 频繁 GC → 堆大小 / 内存泄漏 3) 锁竞争 |
| 内存 OOM | 1) 导出全表 → 用 EasyExcel 分页流式 2) 文件上传内存 3) 缓存未淘汰 |
| 数据库连接耗尽 | HikariCP 默认 10,生产改 50 + 监控 |
| Redis 慢 | 大 Key(`SCAN` 替代 `KEYS *`) / 热 Key 拆分 |

## 六、调试常见 Bug 与定位

### 6.1 后端经典 Bug

#### ❌ 1) 事务不生效

**现象**:方法上加了 `@Transactional` 但部分 SQL 没回滚。

**原因**:
- 自调用(`this.method()` 绕过 Spring 代理)
- 异常被 `try-catch` 吞了
- 非 `public` 方法
- 异常类型不在 `rollbackFor` 默认范围(`RuntimeException` 才会回滚)

**解决**:
```java
// ✅ 正确
@Transactional(rollbackFor = Exception.class, propagation = Propagation.REQUIRED)
public void save() {
    try {
        // ...
    } catch (Exception e) {
        TransactionAspectSupport.currentTransactionStatus().setRollbackOnly();
        throw e;
    }
}
```

#### ❌ 2) 软删除数据查不到

**现象**:明明库里 `deleted=0`,但 `selectById` 返回 null。

**原因**:MyBatis Plus 默认带 `deleted=0` 过滤。

**解决**:用 `Wrapper`:
```java
LambdaQueryWrapper<T> w = Wrappers.<T>lambdaQuery();
w.eq(T::getId, id).eq(T::getDeleted, 0); // 显式带上
// 或
T t = mapper.selectById(id); // 默认过滤 deleted
```

#### ❌ 3) BigDecimal 比较用 ==

```java
new BigDecimal("1.00") == new BigDecimal("1.0")  // ❌ 永远是 false(对象比较)
a.equals(b)                                       // ⚠️ 比较精度(1.0 != 1.00)
a.compareTo(b) == 0                               // ✅ 数值比较
```

#### ❌ 4) 日期时区错乱

**现象**:存进去是 2026-06-16,读出来变成 2026-06-15。

**解决**:
- DB 时区 +8,JDBC URL 加 `serverTimezone=Asia/Shanghai`
- 后端统一用 `LocalDateTime`,序列化用 Jackson `JavaTimeModule` + `WRITE_DATES_AS_TIMESTAMPS=false`
- 前端用 `dayjs` / `date-fns`,不要直接用 `new Date()`

#### ❌ 5) 雪花 ID 精度丢失

**现象**:JS 收到的 19 位 Long(如 `1815648375917510656`)变成 `1815648375917510700`。

**解决**:
- 后端返回前转 String(`@JsonSerialize(using = ToStringSerializer.class)`)或全局配置
- 前端统一按 string 处理,展示再 `Number()` 转

```java
// JacksonConfig
@JsonComponent
public class JacksonConfig {
    @Bean
    public Module longToStringModule() {
        SimpleModule m = new SimpleModule();
        m.addSerializer(Long.class, ToStringSerializer.instance);
        m.addSerializer(Long.TYPE, ToStringSerializer.instance);
        return m;
    }
}
```

### 6.3 数据库 Bug

#### ❌ 死锁

**现象**:`Deadlock found when trying to get lock; try restarting transaction`。

**排查**:
```sql
SHOW ENGINE INNODB STATUS\G
-- 看 LATEST DETECTED DEADLOCK 段
```

**解决**:
- 所有事务按**同一顺序**加锁(如先主表后从表)
- 减少事务粒度
- 加合适索引(行锁 vs 表锁)

#### ❌ 索引失效

| 场景 | 索引是否生效 |
|---|---|
| `WHERE col + 1 = 5` | ❌ |
| `WHERE col = 'x' AND SUBSTR(col, 1, 2) = 'x'` | ❌ 后者函数 |
| `WHERE col LIKE '%x'` | ❌ 前缀通配 |
| `WHERE col IS NULL` | ✅(单独索引) |
| `WHERE col = 1 OR col = 2` | ✅(小范围) |
| `WHERE col IN (...)` | ✅(小范围) |
| 联合索引 `(a,b,c)`,条件 `b=?` | ❌ 缺前缀 |

**排查**:
```sql
EXPLAIN SELECT * FROM inv_stock WHERE product_id = 1;
-- 看 type(ALL=全表扫描, ref/range=走索引)
-- 看 key(实际使用的索引)
```

## 八、问题排查工作流(7 步法)

接到 Bug 按以下顺序排查,平均定位时间 < 10 分钟:

1. **看现象**:错误截图、复现步骤、影响范围
2. **看日志**:`grep <traceId>`,找到 ERROR / WARN
3. **看请求**:Network → 接口返回 / payload 异常
4. **看 SQL**:`p6spy` / `MyBatis log` 看实际执行 SQL,`EXPLAIN` 看索引
5. **看代码**:Git blame 找到相关 Service / Controller
6. **看数据**:`SELECT * FROM table WHERE id = ?`,验证数据
7. **看依赖**:Redis 是否在 / MySQL 主从延迟 / 第三方 API 超时
